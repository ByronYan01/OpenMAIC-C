# OpenMAIC 后端逻辑实现梳理（小白快速上手）

> 目标：你不需要先懂全部前端，只要看这一份就能快速知道后端怎么接请求、怎么调模型、怎么落库（落文件）。

## 1. 后端整体架构（一句话）

这个项目是 **Next.js App Router 的 API Route + 业务服务层 + 文件存储层**：
- API 路由放在 `app/api/*`，负责参数校验、错误返回、调用服务。
- 业务逻辑放在 `lib/server/*`、`lib/generation/*`、`lib/media/*`、`lib/audio/*`。
- 持久化不是数据库，默认是写本地 JSON 文件（`data/` 下）。

## 2. 你先记住 5 条主线

1. 课堂生成有两条链路：
- 交互式链路（前端一步步调 `/api/generate/*`）
- 异步任务链路（后端一次性跑完 `/api/generate-classroom`）

2. 模型解析核心入口：`lib/server/resolve-model.ts`
- 统一处理 `model/apiKey/baseUrl/providerType`。

3. Provider 服务端配置核心：`lib/server/provider-config.ts`
- 配置优先级：客户端传参 > server-providers.yml > 环境变量（按类型映射）。

4. 持久化核心：`lib/server/classroom-storage.ts`
- 课堂、任务状态都写 JSON，使用原子写避免半写入。

5. 安全核心：`lib/server/ssrf-guard.ts`
- 生产环境下对自定义 `baseUrl` 做 SSRF 校验。

## 3. API 分层图（按职责看）

### 3.1 课堂生成相关
- `app/api/generate/scene-outlines-stream/route.ts`
  - 流式生成大纲（SSE）。
- `app/api/generate/scene-content/route.ts`
  - 生成单场景内容。
- `app/api/generate/scene-actions/route.ts`
  - 基于内容生成动作脚本。
- `app/api/generate-classroom/route.ts`
  - 创建异步任务（返回 `jobId`）。
- `app/api/generate-classroom/[jobId]/route.ts`
  - 轮询任务状态和结果。

### 3.2 媒体与语音
- `app/api/generate/image/route.ts`
- `app/api/generate/video/route.ts`
- `app/api/generate/tts/route.ts`
- `app/api/transcription/route.ts`

### 3.3 配置与校验
- `app/api/server-providers/route.ts`
  - 返回服务端可用 provider（不暴露 key）。
- `app/api/verify-model/route.ts`
- `app/api/verify-image-provider/route.ts`
- `app/api/verify-video-provider/route.ts`
- `app/api/verify-pdf-provider/route.ts`

### 3.4 其他能力
- `app/api/parse-pdf/route.ts`（PDF 解析）
- `app/api/web-search/route.ts`（联网检索）
- `app/api/chat/route.ts`（聊天流式输出）
- `app/api/classroom/route.ts`（课堂读写）

## 4. 端到端流程（重点）

### 4.1 交互式生成流程（你在页面上最常用）

1. 前端先调 `/api/generate/scene-outlines-stream`
- 后端做参数解析、模型解析（从 header 取 `x-model/x-api-key/x-base-url`）
- 通过 SSE 分片返回大纲

2. 再调 `/api/generate/scene-content`
- 对某个 outline 生成内容（讲解文本、结构等）

3. 再调 `/api/generate/scene-actions`
- 把内容转成“可播放动作”（说话、切换、互动等）

4. 如果开启媒体/语音
- 调 `/api/generate/image`、`/api/generate/video`、`/api/generate/tts`

这条链路特点：
- 前端体验更快（先看到部分结果）
- 容错细（某一步失败还能继续调整）

### 4.2 异步任务流程（后端一把跑完）

1. `POST /api/generate-classroom`
- 校验 `requirement`
- 生成 `jobId`
- `createClassroomGenerationJob` 写入 queued 状态
- 通过 `after(() => runClassroomGenerationJob(...))` 在后台执行

2. `runClassroomGenerationJob`
- 标记 running
- 调 `generateClassroom`
- 持续写入进度（step/progress/message）
- 最终标记 succeeded / failed

3. `GET /api/generate-classroom/[jobId]`
- 前端轮询查看进度
- done=true 时拿到结果 URL 和课堂 id

这条链路特点：
- 适合外部系统集成、批量任务
- 前端不用手动串每一步

## 5. generateClassroom 后端流水线拆解

文件：`lib/server/classroom-generation.ts`

主步骤（对应 step 字段）：
1. `initializing`：初始化、解析语言模型、校验 API Key
2. `researching`：可选 web-search（Tavily）
3. `generating_outlines`：生成大纲
4. `generating_scenes`：循环生成每个场景的 content + actions
5. `generating_media`：可选图片/视频生成并回填占位符
6. `generating_tts`：可选批量语音
7. `persisting`：写入课堂文件
8. `completed`：完成返回

关键细节：
- agentMode=`generate` 时会先用 LLM 动态生成教师/学生画像。
- 即使媒体/TTS失败，主课堂也尽量继续完成（降级不中断）。

## 6. 任务状态与持久化

### 6.1 任务状态
文件：`lib/server/classroom-job-store.ts`

- 状态机：`queued -> running -> succeeded/failed`
- 每个任务一个 JSON 文件
- 有 per-job 锁，避免并发写乱
- 有“僵尸任务”识别：30 分钟无更新，读时判定为 stale failed

### 6.2 课堂数据
文件：`lib/server/classroom-storage.ts`

- 课堂内容与任务都落在 `data/` 目录
- 写文件使用原子写（先写临时文件再替换）
- `buildRequestOrigin` 用于拼接对外 URL

## 7. 模型与 Provider 配置怎么生效

### 7.1 配置中心
文件：`lib/server/provider-config.ts`

配置源：
1. `server-providers.yml`
2. `.env.local` 映射变量（例如 `OPENAI_API_KEY`）

按能力分区：
- `providers`（LLM）
- `tts`
- `asr`
- `pdf`
- `image`
- `video`
- `web-search`

### 7.2 LLM 解析链
文件：`lib/server/resolve-model.ts`

- 读取请求头：`x-model/x-api-key/x-base-url/x-provider-type`
- 解析 provider/model
- 决定 apiKey/baseUrl/proxy
- 调 `getModel()` 得到可直接调用的模型对象

### 7.3 TTS/ASR 与 Image/Video 的差异（你之前问到的点）

- LLM/TTS/ASR 多数走“OpenAI 风格接口”，在实现层通常有明确 `model` 字段。
- Image/Video 这里走的是“provider 适配器模式”，请求头才可选传 `x-image-model` / `x-video-model`，并不是每个 provider 都强依赖统一 model 参数。
- 所以 UI 上看起来会有“配置方式不完全一致”，本质是后端抽象层设计不同：
  - 语音更像统一协议
  - 图像/视频更像多厂商异构协议

## 8. 错误处理与返回规范

文件：`lib/server/api-response.ts`

- 成功统一 `apiSuccess(...)`
- 失败统一 `apiError(code, status, message, details?)`
- 常见错误：
  - `MISSING_REQUIRED_FIELD`
  - `MISSING_API_KEY`
  - `INVALID_URL`
  - `INTERNAL_ERROR`

这让前端更容易做统一 toast/重试逻辑。

## 10. 你后续排错可直接用的定位法

1. 接口报错先看 route.ts：
- 参数有没有传
- header 有没有传
- 是否被 SSRF 校验拦截

2. 再看 provider-config：
- 该能力类型（tts/asr/image/video）有没有对应 key
- baseUrl 是否走了客户端直传分支

3. 最后看具体 provider 实现：
- 请求 path / body / model 字段是否符合你私域服务
